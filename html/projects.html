Automation of test cases for a web application

Automation of test cases for a web application

A

 

B

RIEF 

O

VERVIEW

In the summer of 2019, I was hired as a software intern for Itiviti India Private Limited, where I worked on two projects. The organization develops software products to help financial institutions worldwide with their trading practices. In an era where the capital markets are highly complex, automated trading operation is a highly complex business, where the lack of testing strategies has a huge impact. Managing test environments, with multiple test components, is costly and resource-intensive both in terms of staffing and technical infrastructure requirements. Firms may end up spending more time maintaining their complex testing environments than performing actual tests. VeriFIX Enterprise is a software application developed by Itviti which helps firms manage their testing environments. This project aimed at automating the test cases for the software application, VeriFIX Enterprise.

P

ROJECT 

P

ROBLEM 

S

TATEMENT

As the application of VeriFIX enterprise is constantly developing, upgrading and adding new features to it, proof testing of the upgrades and newer additions are necessary to find out the bugs and their removal. To test any software, manual testing is performed on the newly upgraded components, which is followed by automated testing of the complete software. For certain components, the manual testing part had been completed, and the test cases needed to be automated to fasten the process of testing in further deployments. The task assigned to me was to validate the working of a filter criteria button in the web application and generate test cases for the same.

T

OOLS 

U

SED

Developing Environment Eclipse
Programming Language Java
Testing Framework JUnit
Automation Driver Selenium Web Driver
Other Tools Git

A

CTIONS 

T

AKEN

To test the filter criteria button, the first step was logging in to the web application. To automate this step, the Selenium web driver was used to log in and perform all the navigational tasks that had to be performed before the actual test started. The code for performing all these tasks was kept in the proper annotation of the JUnit Test Case so that these tasks would run at the beginning. Before applying the filter, all the messages were extracted and those which had the desired filter present in them were counted. The count was stored. The next step was to apply the filter and count the total number of messages present. If this number matched with the number stored, this part of the test would be a success, else it would fail and indicate that the filter criteria button wasn't working properly. An additional condition was added to further validate the button, which was to check if every message in the message box after application of filter criteria contained the desired filter or not. The first condition ensured that no messages were missed out after the application of the filter, while the second condition ensured that no messages that shouldn't be present appeared in the message box. If both the conditions passed successfully, the test passed, indicating that the filter criteria button was working properly. If any of the above conditions failed, the test would fail and indicate some error in the working of the filter criteria button.

  • Learnt a new programming language Java and a testing framework JUnit associated with it
  • Learnt the usage of Selenium Web Driver to extract data out from the web

Automation of daily checks for a package

Automation of daily checks for a package

A

 

B

RIEF 

O

VERVIEW

In the summer of 2019, I worked as a software intern for Itiviti India Private Limited, where I worked on two projects. The organization develops software products to help financial institutions worldwide with their trading practices. One of their products was the Itiviti Analyst package. It is a package consisting of a suite of apps, sessions, disks as well as databases which ensures that the various trading practices followed by its various clients comply with various trade regulations. Morning checks are a series of checks being done daily to ensure the proper implementation of the Itiviti Analyst package. This project aimed at automating the manual process of morning checks.

P

ROJECT 

P

ROBLEM 

S

TATEMENT

The Itiviti Analyst package is implemented on the client-side of the server. Post-implementation, the team ensures that all the apps run correctly, checks whether all the sessions are appropriately configured and verifies if the data collection is happening correctly. To ensure this, a series of checks were done daily, which are known as morning checks. The entire process of morning checks consists of 10 components, which were performed one after the other. This was followed by the preparation of a report that contained the results and summary of the checks. This report was sent to the client via email. This process of performing the checks and preparing the reports was being carried out manually for every client. Only the process of sending email had already been automated. The entire process, from performing the check to sending the email used to take around 1 to 1.5 hours per client. The task assigned to me was to automate the processes of performing the checks and preparing the reports so that it could be seamlessly integrated with the existing script which sent the email.

T

HE 

M

ANUAL 

P

ROCESS

For performing the checks, the tester had to login into the client-side of the server through a remote access connection. The server was a Linux server and the software PuTTY was used for this purpose. After having remote access to the server, the tester had to login into the database containing the details of all the package components configured for the client. The status of all the components can be known from the database. Then, corresponding to each of the 10 components of the morning checks, a series of commands were fired from the terminal. The output from every command was collected manually and stored. These outputs were combined and accordingly processed to generate the result of the particular component. The results were integrated into a file to prepare the report.

A

CTIONS 

T

AKEN
Selection of a suitable library

The basic idea behind the automation of morning checks was to replicate the process being carried out manually. Since the manual process included the firing of commands into the Linux terminal, a library had to be chosen which could fire commands into the terminal from the script itself and collect the output. Various Python libraries are available which allow the user to interact with the command line. Some notable ones among them are Python Fire, subprocess, threading, multithreading and multiprocessing libraries. For choosing the appropriate library, the following points had to be considered:

  • The library must allow the creation and running of a new process from the script itself
  • The library must check the output of the process i.e., whether the command successfully ran or not
  • The library must be able to collect the output of the command displayed on the terminal and save it in a file
  • The library must be present in Python 2.7, which was the version of Python being used on the server at that time

Considering these points, the subprocess library was chosen as it satisfied all of them.

Automating the components of the checks

After a suitable library was chosen, the next task was to write a script to automate each component of the morning checks. For every component, one or more commands were fired into the terminal using the subprocess library. Depending upon the commands and the requirements, either the output of the process (whether the process ran successfully or not) or the output displayed by the command on the terminal was collected. This output was stored in a text file and the required processing related to the component of the check was performed.

Preparation of morning checks report

The report of morning checks was prepared in two formats. One was a regular .txt file and the other one was a .html file. At the beginning of the script, an empty .txt/.html file was created. After processing the outputs from the terminal, the results were appended to the .txt file. In the case of the .html file, appropriate tags were inserted at appropriate locations to ensure the proper formatting of the report.

Handling exceptions during runtime

The biggest issue during the automation of morning checks was the handling of exceptions. The entire process of morning checks was being carried out on a real-time operating server. The server being dynamic was bound to cause misfunctioning of commands at certain times. All the commands in the script were being fired in series. Hence, the termination or misfunctioning of a single command would lead to the termination of the entire script. To avoid this, exception handling was needed. This was achieved by carefully examining the piece of code for possible sources where termination of the script could take place. To handle these exceptions, a try-except block was used for every possible case. The except block was carefully designed to ensure that the script doesn't terminate due to the non-functioning of commands.

Ensuring robustness of the created script

The most important part of the project was to ensure the robustness of the script. This was done to ensure that the same script could be used for different clients by making minimal changes to it. The different variables for a client were stored in different lists, each corresponding to a different command, at the top of the script. While execution, the script would take values from the list, insert them at the appropriate position in the command and then fire it. Hence, the process of modifying the script for different clients reduced down to suitably manipulating the different lists for a particular client. The script was designed to take care of the rest.

K

EY 

C

ONTRIBUTIONS
  • Created a Python script that performs all the components of the morning checks and generates the report in .txt format.
  • Created another Python script that performs the morning checks and generates the reports in .html format. This report could be easily integrated with another existing Python script which would send it directly to the client via an email. Thus, firing a single command to run the script would perform all the morning checks, generate the report and send it to the client.
  • Ensured robustness of the created script, thereby allowing the same script to prepare the reports for multiple clients by fine-tuning it.
  • Reduced the time taken for preparation of morning checks reports significantly from 1 hour to a worst-case of less than 2 minutes per client. The worst-case time was due to the presence of such commands which on receiving no output for the terminal would continue to run for 30 seconds. On an average case, the script was competent enough to prepare the reports in less than 30 seconds.
  • Ensured that the created reports do not eat up the memory by suitably deleting old reports after a few days. The number of days for which the report has to be kept can be specified by the user. On execution, the script would check for old morning checks reports, delete them and then perform the checks.

Dynamics and optimization of conventional distillation columns

Dynamics and optimization of conventional distillation columns

A

 

B

RIEF 

O

VERVIEW

During my bachelor's project in 2019-20, I worked with a team of researchers from Energy and Process Engineering Laboratory, IIT Kharagpur. The team of researchers were working on various types of industrial distillation processes and working on ways to make the process more energy-efficient. It is a well-established fact that for multicomponent distillation involving n components, at least n-1 distillation columns are needed to achieve the desired degree of purification. Hence, a ternary mixture requires at least two distillation units for its separation. A single distillation column itself demands energy in the range of 1-100 MW. Thus, the energy requirement for the separation of a ternary mixture itself using distillation is enormous. To add to that, 95% of liquid separation processes use distillation as their primary method of separation, which are mostly conventional distillation units. Hence, the field of distillation presents itself with a high potential for possible energy savings. This project aimed at modelling a distillation system and obtaining an optimal operating configuration for a conventional distillation sequence involving two distillation units in series. The results from this project served as a minimum benchmark with which the results from other modifications of distillation units were compared.

O

BJECTIVES

The main objectives of the project were threefold:

  • To develop a model of the conventional distillation sequence in series and integrate it with MATLAB
  • To develop a generalised multi-objective optimization problem to be solved across different systems
  • To use a non-dominated sorting genetic algorithm (NSGA-II) to obtain an optimal solution for the system

Two ternary systems, Benzene - Toulene - Xylene (BTX) and Benzene - Toluene - Etyhlbenzene (BTE), with fixed feed conditions were chosen for the study. NSGA-II was employed to find an optimal solution.

T

OOLS 

U

SED

Programming Language MATLAB
Modelling Tool Aspen Plus
Interfacing Server ActiveX
Optimzation Algorithm NSGA - II

G

ENERALIZED 

O

PTIMIZATION 

P

ROBLEM

The following generalised optimization problem was formulated, which was solved using NSGA-II using the three step method described below.

Minimize Capital Cost
Minimize Operating Cost
Minimize CO2 emission
Maximize the top component recovery
Maximize the middle/bottom component recovery
subject to purity constraints

A

CTIONS 

T

AKEN
Modelling the system and interfacing it

The conventional distillation sequence was modelled using the direct sequence in Aspen. Two different but non-independent RadFrac columns were employed in Aspen to model the system using the rigorous method of simulation. The system modelled in Aspen was interfaced with MATLAB using the ActiveX server. MATLAB was used to generate all the inputs, which were sent to Aspen via the ActiveX server. The simulations were run and completed in Aspen, the results were prepared and MATLAB was notified about the completion of the simulation. The outputs were collected from Aspen and then used further.

Multi-objective optimization

To determine the decision variables which influence the objectives, sensitivity analysis was carried out. Different inputs were chosen and their correlation with all the objectives was found. The data points for determining the correlation coefficient were also generated through rigorous simulation in Aspen. From this, those variables that had a significant influence on any of the objective functions were chosen and the rest were discarded. The results after simulation were fed as input to the NSGA-II algorithm to run the optimization and generate a pareto-optimal front.

Selection of one-optimal point

NSGA-II gives a set of optimal solutions, none of which is more dominating than the other. However, the determination of one condition out of the many generated remains an issue. Many methods have been suggested to do so. The technique for order of preference by similarity to ideal solution (TOPSIS) is one of the most straightforward and most practical techniques for the identification of a truly optimal solution from the set of non-dominated solutions and was used to select one optimal point from the set of solutions.

PTIMIZATION

F

UTURE 

W

ORK

This project dealt with the multi-objective optimization of a conventional distillation sequence containing two distillation units in series. However, various alternatives have been suggested where the process can be carried out in a single column, thus leading to significant savings in capital cost, operating cost, as well as space. The steps post this work was to carry out the same process for one of the alternatives, the dividing wall column.

Multi-objective optimization of dividing wall columns

Multi-objective optimization of dividing wall columns

A

 

B

RIEF 

O

VERVIEW

During my bachelor's project in 2019-20, I had worked on a project to optimize the configuration of a conventional distillation sequence involving two columns in series. The project on dividing wall columns (DWC) is an extension of that project. The dividing wall column is an intensified equipment that integrates the two columns of a conventional distillation column into a single shell. Hence, a ternary mixture separation is possible through a single column, compared to the conventional distillation sequence that requires two columns. Hence, the presence of a single column, reboiler and condenser allow it to deliver a large amount of savings in the operating and capital costs. Installing a DWC over two conventional units can lead to a huge reduction in the installation space as well. This project aimed at evaluating the potential savings if the conventional distillation sequence (CDS) was replaced with a dividing wall column.

O

BJECTIVES

The main objectives of the project were following:

  • To develop a model of the dividing wall column in Aspen and integrate it with MATLAB
  • To use a non-dominated sorting genetic algorithm (NSGA-II) to obtain an optimal solution for the same systems using the same generalized optimization problem as in CDS.

The same two ternary systems, Benzene - Toulene - Xylene (BTX) and Benzene - Toluene - Etyhlbenzene (BTE), with the feed conditions similar to the previous study were chosen here as well.

Optimzation Algorithm NSGA - II

The technique for order of preference by similarity to ideal solution (TOPSIS) is one of the most straightforward and most practical techniques for the identification of a truly optimal solution from the set of non-dominated solutions and was used to select one optimal point from the set of solutions.

RIEF 

O

VERVIEW

During my bachelor's project in 2019-20, I had worked on a project to optimize the configuration of a conventional distillation sequence involving two columns in series. The project on dividing wall columns (DWC) is an extension of that project. The dividing wall column is an intensified equipment that integrates the two columns of a conventional distillation column into a single shell. Hence, a ternary mixture separation is possible through a single column, compared to the conventional distillation sequence that requires two columns. Hence, the presence of a single column, reboiler and condenser allow it to deliver a large amount of savings in the operating and capital costs. Installing a DWC over two conventional units can lead to a huge reduction in the installation space as well. This project aimed at evaluating the potential savings if the conventional distillation sequence (CDS) was replaced with a dividing wall column.

O

BJECTIVES

The main objectives of the project were following:

  • To develop a model of the dividing wall column in Aspen and integrate it with MATLAB
  • To use a non-dominated sorting genetic algorithm (NSGA-II) to obtain an optimal solution for the same systems using the same generalized optimization problem as in CDS.

The same two ternary systems, Benzene - Toulene - Xylene (BTX) and Benzene - Toluene - Etyhlbenzene (BTE), with the feed conditions similar to the previous study were chosen here as well.
